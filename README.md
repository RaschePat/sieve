# SIEVE

> 체로 거르듯 — **단순 작업을 걸러내 로컬 LLM으로, 복잡한 작업만 클라우드로** 보내 API 토큰 비용을 줄이는 OpenAI 호환 프록시.

기존 코드에서 **base URL만** 이 라우터로 바꾸면, 요약·번역·분류 같은 정형 작업은 로컬 모델(MLX/Ollama 등)이 처리하고, 코드 리뷰·디버깅·설계처럼 추론이 필요한 작업만 Claude로 보냅니다. 모든 요청의 라우팅 결정·토큰·비용·품질을 기록해 **"얼마나 절약했는지"를 숫자로 증명**합니다.

```
클라이언트 (base_url = http://localhost:8400/v1)
      │
      ▼
┌─────────────────────────────────────────────┐
│  SIEVE (FastAPI)                             │
│                                              │
│  ① 분류기 (휴리스틱)  ── 작업 난이도 판정     │
│        │                                     │
│        ├─ 정형 작업 ──▶ 로컬 LLM (oMLX/Qwen2.5 7B)
│        │                예: 요약·번역·분류·추출·포맷
│        │                                     │
│        └─ 고난도 작업 ─▶ 클라우드 (Claude Opus 4.8)
│                         예: 코드리뷰·디버깅·설계
│                                              │
│  ② 폴백: 로컬 실패 시 자동으로 클라우드 재시도 │
│  ③ 로깅: 모든 요청을 SQLite에 기록           │
│  ④ 품질검증: 로컬 응답 샘플링 → Claude가 채점 │
└─────────────────────────────────────────────┘
      │
      ▼
  /stats · /dashboard · scripts/report.py
```

## 왜 만들었나

LLM을 실무에 쓰면 비용 대부분이 **단순 반복 작업**(요약, 태깅, 포맷 변환)에서 나옵니다. 이런 작업은 7B급 로컬 모델로도 충분한데, 전부 클라우드 API로 보내면 토큰 비용이 그대로 쌓입니다. SIEVE는 그 낭비를 라우팅으로 제거하고, **절감 효과를 계측·벤치마크·품질검증 세 축으로 증명**하는 것이 목표입니다.

## 핵심 기능

| 기능 | 설명 |
|---|---|
| **OpenAI 호환 API** | `/v1/chat/completions` — 기존 코드에 base URL만 교체 |
| **휴리스틱 라우팅** | 작업 유형·입력 길이·코드 포함 여부로 local/cloud 결정. 모든 결정에 사유 기록 |
| **캐스케이드 폴백** | 로컬 모델이 실패(빈 응답·에러)하면 클라우드로 자동 재시도 |
| **비용 계측** | 요청마다 `{경로, 토큰, 비용, 절약액, 지연}`을 SQLite에 기록 |
| **품질 검증** | 로컬 응답을 확률 샘플링(기본 10%)해 Claude가 1\~5점 채점 (LLM-as-judge) |
| **실시간 대시보드** | `/dashboard` — 로컬 처리율·절약액·품질점수를 10초마다 갱신 |

## 빠른 시작

### 1. 로컬 LLM 서버 준비

OpenAI 호환 엔드포인트를 제공하는 로컬 서버면 무엇이든 됩니다.

- **oMLX** (이 프로젝트 기본): `http://localhost:8000/v1`, 모델 `Qwen2.5-7B-Instruct-4bit`
- **LM Studio**: `http://localhost:1234/v1`
- **Ollama**: `http://localhost:11434/v1`
- **vLLM / llama.cpp server** 등

### 2. 설치 & 설정

```bash
python3 -m venv .venv && source .venv/bin/activate
pip install -r requirements.txt

cp .env.example .env
# .env 에서 LOCAL_BASE_URL / LOCAL_API_KEY / ANTHROPIC_API_KEY 설정
export ANTHROPIC_API_KEY=sk-ant-...        # 콘솔 발급 API 키 (구독 계정과 별개)
```

### 3. 실행

```bash
uvicorn app.main:app --port 8400
```

### 4. 호출

```bash
# 요약 → 로컬로 라우팅
curl http://localhost:8400/v1/chat/completions \
  -H "Content-Type: application/json" \
  -d '{"messages":[{"role":"user","content":"다음을 한 줄로 요약: ..."}]}'

# 응답의 router 필드로 라우팅 결과 확인
# "router": {"route":"local","reason":"정형 작업 감지: 요약","cost_usd":0,"saved_usd":0.0016, ...}
```

기존 SDK 코드라면 base URL만 바꾸면 됩니다:

```python
from openai import OpenAI
client = OpenAI(base_url="http://localhost:8400/v1", api_key="unused")
```

`x-route: local` 또는 `x-route: cloud` 헤더로 라우팅을 강제할 수도 있습니다.

## 벤치마크

동일한 12개 태스크(요약·번역·분류·추출·코드리뷰·디버깅·설계 등)를 두 모드로 실행해 비용을 비교합니다.

```bash
python benchmark/run.py              # 클라우드-only vs 하이브리드 비교
python benchmark/run.py --hybrid-only  # 클라우드 키 없이 하이브리드만
```

### 실측 결과 (12개 태스크, Opus 4.8 vs Qwen2.5-7B-4bit/oMLX)

| 지표 | 클라우드-only | 하이브리드 |
|---|---|---|
| 총 비용 (USD) | $0.10276 | $0.06173 |
| 로컬 처리 건수 | 0 / 12 | **8 / 12** |
| 클라우드 호출 건수 | 12 | 4 |
| 벽시계 시간 (s) | 65.0 | 120.8 |

**절감액: $0.04104 (39.9% 절감)** — 12건 중 8건(요약·번역·분류·추출·포맷)을 로컬에서 처리.

#### 드러난 트레이드오프 (정직하게)

1. **비용 ↓ 40%, 지연 ↑ 약 2배.** 로컬 7B(Apple Silicon, 4bit)가 클라우드보다 느립니다. 비용에 민감하고 지연에 관대한 배치/비동기 워크로드에 적합하고, 실시간 응답이 중요하면 로컬 하드웨어를 키우거나 라우팅 임계값을 조정해야 합니다.
2. **로컬 품질 한계 — LLM-as-judge 평균 3.25/5.** `Qwen2.5-7B-Instruct-4bit`가 **한국어 요청에 중국어를 섞는** 문제가 관찰됐습니다(요약·번역에서 1\~2점). 분류·추출·정보성 작업은 4\~5점으로 양호. → 개선 방향: 한국어 강한 모델 교체, 4bit→8bit 양자화, 또는 로컬 호출 시 "반드시 한국어로 답하라" 시스템 프롬프트 주입.

> 이 두 한계는 숨기지 않고 리포트(`fallback` 집계 + judge 점수)로 드러나게 설계했습니다. "로컬이 항상 옳다"가 아니라 "어디까지 로컬에 맡길 수 있는가"를 데이터로 보여주는 것이 목표입니다.

### 발견 → 개선: 한국어 가드 (A/B 검증)

위에서 드러난 **중국어 혼용 문제**를 로컬 호출 시 언어 가드 시스템 프롬프트로 해결했습니다. 핵심은 "한국어로 답하라"고 무작정 강제하지 않고 **번역·작성 요청은 예외**로 둔 것 — 그래야 "영어로 번역해줘" 같은 요청을 망치지 않습니다.

측정은 주관적인 judge 대신, **응답에 섞인 중국어 한자(漢字) 개수**라는 결정론적 지표를 씁니다(한글과 한자는 유니코드 블록이 다름). `python benchmark/korean_guard_ab.py`:

| 태스크 | 유형 | 한자 OFF | 한자 ON |
|---|---|---|---|
| trans-02 | 번역(한→한) | 84 | **0** |
| ext-01 | 추출 | 56 | **0** |
| fmt-01 | 포맷변환 | 20 | **0** |
| trans-01 | 번역(한→영) | 0 | 0 |
| (그 외 4건) | | 0 | 0 |

- **중국어 혼용 응답: 3/8 → 0/8**
- **총 한자: 160 → 0 (−100%)**
- `trans-01`(영어로 번역) 은 0→0 유지 — 가드가 번역 작업을 침해하지 않음을 확인

`LOCAL_KOREAN_GUARD=0` 으로 끌 수 있고, 기본은 켜져 있습니다.

## 운영 리포트

```bash
python scripts/report.py > REPORT.md
```

DB에 쌓인 모든 요청을 집계해 로컬 처리율, 절감율, 라우팅 사유별 분포, LLM-as-judge 품질 점수를 마크다운으로 출력합니다.

## 측정 방법론 (정직한 한계)

신뢰성을 위해 측정의 전제와 한계를 명시합니다.

1. **절약액은 추정치입니다.** 로컬 처리 시 "같은 토큰량을 클라우드로 보냈다면 들었을 비용"을 절약액으로 계산합니다. 단 **로컬 모델과 Claude의 토크나이저가 달라** 토큰 수가 정확히 일치하지 않으므로 근사치입니다. 방향성과 규모를 보는 용도입니다.
2. **클라우드 단가**는 `app/config.py`의 `PRICING` 표(2026-06 기준)를 사용합니다. 프롬프트 캐싱은 계산에 반영하지 않습니다(보수적 추정).
3. **품질 점수**는 클라우드 모델(Claude)이 채점한 상대 평가입니다. 절대적 정답 검증이 아니라 "로컬 응답이 요청을 충족했는가"의 프록시 지표입니다.
4. **로컬 모델의 한계도 기록합니다.** 폴백이 발생한 요청(로컬 실패 → 클라우드)은 `fallback` 플래그로 별도 집계되어, 어떤 작업에서 로컬이 부족했는지 드러납니다.
5. **분류기는 휴리스틱이라 한계가 있습니다.** 라우터가 전체 메시지(system·user 등)를 합쳐 패턴 매칭하므로, system 프롬프트나 인용문에 우연히 들어간 키워드가 라우팅을 흔들 수 있습니다. 현재는 "애매하면 클라우드"로 보수적으로 처리하며, 다음 단계는 경량 분류 모델로의 교체입니다.

## 프로젝트 구조

```
app/
  main.py            # FastAPI — /v1/chat/completions, /stats, /dashboard
  router.py          # 휴리스틱 분류기 (local vs cloud 결정 + 사유)
  config.py          # 모델 설정 + Claude 단가표
  db.py              # SQLite 로깅 & 집계
  judge.py           # LLM-as-judge 품질 채점
  providers/
    local.py         # OpenAI 호환 로컬 서버 호출 (oMLX/LM Studio/Ollama)
    cloud.py         # Anthropic Claude API 호출 (adaptive thinking)
benchmark/
  tasks.jsonl        # 벤치마크 태스크 셋
  run.py             # 클라우드-only vs 하이브리드 비교
  korean_guard_ab.py # 한국어 가드 A/B (중국어 혼용 before/after)
scripts/
  report.py          # DB → 마크다운 리포트
```

## 모델 선정 메모

- **로컬: Qwen2.5 7B (4bit, MLX/oMLX)** — Apple Silicon에서 MLX 가속으로 동작. 분류·추출·정보성 작업은 양호(judge 4\~5점)하나, **한국어 생성에서 중국어 혼용 문제 관찰**(judge 1\~2점, 실측 평균 3.25/5). → **언어 가드 프롬프트로 해결**: 중국어 혼용 응답 3/8 → 0/8 (위 A/B 참조). 추가로 한국어 강한 모델 교체나 8bit 양자화로도 보완 가능.
- **클라우드: Claude Opus 4.8** — 고난도 추론/코드 작업의 품질 기준점. 단순 작업을 걸러내고 남은 "진짜 어려운 것"만 받으므로 비싼 모델을 써도 총비용이 낮아짐.
- **judge: Claude Haiku 4.5** — 채점은 단순 작업이라 최저가 모델로 충분. judge 자체가 비용을 키우지 않도록 의도적으로 분리(8건 채점에 1센트 미만).
